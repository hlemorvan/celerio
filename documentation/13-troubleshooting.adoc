[[troubleshooting]]
= 13. Dépannage

== Le code généré ne compile pas

Vous pouvez rencontrer une situation où le code Java généré ne compile pas. Pour ne pas perdre trop de temps, si vous savez comment corriger l'erreur de compilation, vous devriez prendre le contrôle total sur le fichier généré qui ne compile pas (voir <<code-modification>>) et corriger l'erreur manuellement.

Signalez également le problème à link:https://github.com/jaxio/celerio/issues[GitHub Issues] pour qu'il soit corrigé dans une prochaine version de Celerio.

== Problèmes spécifiques à Java 25 + Spring Boot 4

[[java-modules]]
=== Modules Java (JPMS) et accès réflexif

Java 25 peut bloquer certains accès réflexifs utilisés par des bibliothèques anciennes. Si vous rencontrez des `InaccessibleObjectException` ou `IllegalAccessException`, ajoutez un fichier `.mvn/jvm.config` à la racine du projet :

[source]
----
--add-opens java.base/java.lang=ALL-UNNAMED
--add-opens java.base/java.util=ALL-UNNAMED
--add-opens java.base/java.lang.reflect=ALL-UNNAMED
----

=== Spring Boot 4 : port déjà utilisé

Si vous obtenez `Port 8081 was already in use`, vérifiez quel processus occupe le port :

[source,bash]
----
lsof -i :8081
# ou
ss -tlnp | grep 8081
----

Pour arrêter l'ancien processus :

[source,bash]
----
kill <PID>
----

=== Spring Security 6+ : `IllegalArgumentException: each password must have a password encoding prefix`

Spring Security 6+ requiert que chaque mot de passe stocké ait un préfixe d'encodage. Pour des mots de passe en clair (développement uniquement), utilisez le préfixe `{noop}` :

[source,sql]
----
-- CORRECT pour développement
INSERT INTO "USER" (..., password, ...) VALUES (..., '{noop}admin', ...);

-- INCORRECT : manque le préfixe
INSERT INTO "USER" (..., password, ...) VALUES (..., 'admin', ...);
----

CAUTION: Le préfixe `{noop}` signifie « pas d'encodage ». Ne jamais l'utiliser en production. Utilisez BCrypt (`{bcrypt}`) ou Argon2 (`{argon2}`) à la place.

=== Jakarta EE : `ClassNotFoundException: javax.persistence.*`

Si vous obtenez des `ClassNotFoundException` pour des classes `javax.persistence.*`, `javax.servlet.*` ou `javax.inject.*`, c'est que le code utilise encore les anciens noms de namespace. Depuis Jakarta EE 9+, tous ces packages sont renommés en `jakarta.*`.

Solution rapide pour les fichiers générés :

[source,bash]
----
find src -name "*.java" -exec sed -i \
  's/javax\.persistence\./jakarta.persistence./g;
   s/javax\.validation\./jakarta.validation./g;
   s/javax\.inject\./jakarta.inject./g;
   s/javax\.servlet\./jakarta.servlet./g' {} \;
----

=== Hibernate Validator 8 : `NotEmpty` et `Email` non trouvés

Hibernate Validator 8 a supprimé `org.hibernate.validator.constraints.NotEmpty` et `org.hibernate.validator.constraints.Email`. Ces annotations sont maintenant dans `jakarta.validation.constraints.*` :

[source,java]
----
// AVANT (Hibernate Validator < 8)
import org.hibernate.validator.constraints.NotEmpty;
import org.hibernate.validator.constraints.Email;

// APRÈS (Hibernate Validator 8+)
import jakarta.validation.constraints.NotBlank;  // équivalent à @NotEmpty
import jakarta.validation.constraints.Email;
----

=== Swagger UI "Failed to load remote configuration"

Si Swagger UI affiche cette erreur, c'est que les endpoints `/v3/api-docs/**` sont protégés par Spring Security. Assurez-vous que `WebSecurityCustomizer` les ignore :

[source,java]
----
@Bean
public WebSecurityCustomizer webSecurityCustomizer() {
    return web -> web.ignoring()
        .requestMatchers("/v3/api-docs/**", "/swagger-ui/**", "/swagger-ui.html");
}
----

=== H2 2.x : `USER` est un mot réservé

H2 2.x a ajouté `USER` comme mot réservé SQL. Si vous avez une table nommée `USER`, mettez-la entre guillemets doubles dans vos scripts SQL :

[source,sql]
----
-- H2 2.x : guillemets doubles obligatoires
CREATE TABLE "USER" (
    id int not null GENERATED BY DEFAULT AS IDENTITY,
    -- ...
);

-- Et dans les requêtes
SELECT * FROM "USER" WHERE login = 'admin';
----

=== H2 2.x : `IDENTITY` dépréciée

H2 2.x a dépréciée la syntaxe `IDENTITY` au profit de `GENERATED BY DEFAULT AS IDENTITY` :

[source,sql]
----
-- H2 1.x (OBSOLÈTE)
id int not null IDENTITY,

-- H2 2.x (CORRECT)
id int not null GENERATED BY DEFAULT AS IDENTITY,
----

=== Celerio ne génère pas les entités (seulement `Identifiable.java`)

Si Celerio ne génère que `Identifiable.java` dans le package domain, la base de données est probablement vide au moment de l'extraction. Vérifiez :

1. Que le profil `db` a bien été activé pour créer la base AVANT `metadata` :
+
[source,bash]
----
mvn -Pdb,metadata,gen generate-sources
----

2. Que l'URL JDBC dans `pom.xml` pointe vers la bonne base :
+
[source,xml]
----
<jdbc.url>jdbc:h2:${project.build.directory}/db/angulardb</jdbc.url>
----

3. Que `celerio-maven-plugin.configuration` pointe vers le bon fichier XML :
+
[source,xml]
----
<celerio-maven-plugin.configuration>
    ../quickstart-conf/celerio-maven-plugin.xml
</celerio-maven-plugin.configuration>
----

=== Collision dans pom.xml

Si votre `pom.xml` est remplacé par une ancienne version après régénération, c'est que Celerio a détecté une modification manuelle et a placé sa version dans le dossier de collision `target/celerio-maven-plugin/collisions/`. Vérifiez ce dossier et fusionnez manuellement les différences.

== Velocity 2.x : `$velocityCount has not been set`

Si vous obtenez l'erreur `Velocitycount has not been set` ou `velocityHasNext has not been set`, vos templates utilisent encore les anciennes variables Velocity 1.x. Voir <<velocity2-changes>>.

[source,bash]
----
# Corriger automatiquement dans les templates
grep -r "\$velocityCount\|\$velocityHasNext" pack-angular/
# Puis remplacer manuellement (voir chapitre 12)
----

== Maven ne trouve pas celerio-maven-plugin

Si Maven ne trouve pas le plugin Celerio 4.0.24-SNAPSHOT, assurez-vous de l'avoir installé localement :

[source,bash]
----
export JAVA_HOME=/home/herve/jdk-25.0.2+10
cd /home/herve/celerio
mvn install -DskipTests
----

NOTE: Vous devez rebuilder **tous les modules** (pas seulement `celerio-engine`) car `celerio-maven-plugin` dépend de `celerio-engine`.

== `JAVA_HOME` non défini

Si vous obtenez des erreurs liées à la version Java, vérifiez que `JAVA_HOME` pointe vers JDK 25 :

[source,bash]
----
export JAVA_HOME=/home/herve/jdk-25.0.2+10
export PATH=$JAVA_HOME/bin:$PATH
java -version
----
