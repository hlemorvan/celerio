[[conventions]]
= 9. Conventions et intégration

Celerio a des conventions intégrées. Quand ces conventions sont suivies, Celerio génère des fonctionnalités spécifiques.

Par exemple, en suivant simplement des conventions de nommage de colonnes, vous pouvez vous appuyer sur Celerio pour générer tout le code d'infrastructure et la configuration qui vous permettront de gérer l'upload et le download de fichiers dans votre application web, de manière optimale.

[[pk-conventions]]
== Conventions de clé primaire

[[numerical-pk]]
=== Clés primaires numériques

Chaque colonne de clé primaire numérique est mappée avec les annotations `@GeneratedValue` et `@Id`.

IMPORTANT: Si votre base de données ne supporte pas les colonnes d'identité, vous devriez créer la séquence `hibernate_sequence`. Référez-vous à la documentation Hibernate pour des alternatives avancées. Pour H2 2.x, utilisez `GENERATED BY DEFAULT AS IDENTITY` (et non `IDENTITY`).

[[pk-32-chars]]
=== Clés primaires de 32 caractères

Par convention, pour toutes les clés primaires `char(32)`, Celerio mappe la colonne avec les annotations suivantes :

[source,java]
----
@GeneratedValue(generator = "strategy-uuid")
@GenericGenerator(name = "strategy-uuid", strategy = "uuid")
@Id
----

Comme cela utilise le `UUIDHexGenerator` d'Hibernate, aucune séquence n'est nécessaire pour ces clés primaires.

[[pk-36-chars]]
=== Clés primaires de 36 caractères

Par convention, pour toutes les clés primaires `char(36)`, Celerio mappe la colonne avec :

[source,java]
----
@GeneratedValue(generator = "strategy-uuid2")
@GenericGenerator(name = "strategy-uuid2", strategy = "uuid2")
@Id
----

[[other-pk]]
=== Autres clés primaires

Pour une clé primaire `char(x)` où x est différent de 32 ou 36, Celerio mappe la colonne avec un générateur `assigned` — vous devez fournir manuellement la valeur de la clé primaire.

[[spring-security-integration]]
== Intégration Spring Security

Le service `UserDetailsServiceImpl` généré est le lien entre le monde Spring Security et les informations d'identification de votre utilisateur.

L'implémentation générée varie selon que vous suivez ou non les conventions Celerio.

Si vos informations d'identification sont stockées dans votre base de données, assurez-vous que l'entité jouant le rôle d'entité `account` a les propriétés attendues par Celerio (ex : login & password).

Une fois que Celerio a trouvé votre entité account, il recherche ses associations many-to-many. S'il trouve une association many-to-many dont l'entité cible joue également le rôle d'entité `Role` (ex : elle a une propriété 'authority'), il l'utilise.

NOTE: Les entités 'Account' et 'Role' n'ont pas besoin de s'appeler Account et Role.

[[account-entity]]
=== Entité Account

Par convention, Celerio considère qu'une entité est l'entité `Account` si elle a les propriétés suivantes :

[cols="2,1,1,3", options="header"]
|===
|Propriété attendue
|Type
|Requis ?
|Description

|username, login, userName, identifiant, email, emailAddress, mail
|String
|Oui
|Login utilisé par l'utilisateur final pour s'authentifier

|password, pwd, passwd, motDePasse
|String
|Oui
|Mot de passe utilisé pour l'authentification

|enabled, isEnabled
|Boolean
|Non
|Seuls les utilisateurs activés (enabled == true) peuvent se connecter
|===

Exemple SQL compatible avec les conventions Celerio (H2 v2) :

[source,sql]
----
CREATE TABLE "USER" (
    id       int not null GENERATED BY DEFAULT AS IDENTITY,
    login    varchar(100) not null,
    password varchar(100) not null,
    email    varchar(100),
    is_enabled boolean not null default true,
    version  int default 0,
    constraint user_unique_1 unique (login),
    primary key (id)
);
----

IMPORTANT: Depuis Spring Security 6+, les mots de passe en clair doivent être préfixés par `{noop}` dans la base de données. Par exemple :

[source,sql]
----
INSERT INTO "USER" (id, login, password, email, is_enabled, version)
  VALUES (-1, 'admin', '{noop}admin', 'admin@example.com', true, 1);
INSERT INTO "USER" (id, login, password, email, is_enabled, version)
  VALUES (-2, 'user',  '{noop}user',  'user@example.com',  true, 1);
----

NOTE: Si aucune entité `Account` n'est trouvée, le `UserDetailsServiceImpl` retourne simplement des utilisateurs et rôles codés en dur (`admin`/`admin`, `user`/`user`).

[[role-entity]]
=== Entité Role

Par convention, Celerio considère qu'une entité est l'entité `Role` si elle a la propriété suivante :

[cols="2,1,1,3", options="header"]
|===
|Propriété attendue
|Type
|Requis
|Description

|authority, roleName, role, nameLocale
|String
|Oui
|Nom du rôle, par exemple : ROLE_USER, ROLE_ADMIN
|===

Exemple SQL complet respectant les conventions Celerio (H2 v2) :

[source,sql]
----
CREATE TABLE ROLE (
    id        int not null GENERATED BY DEFAULT AS IDENTITY,
    role_name varchar(100) not null,
    constraint role_unique_1 unique (role_name),
    primary key (id)
);

CREATE TABLE USER_ROLE (
    user_id int not null,
    role_id int not null,
    constraint user_role_fk_1 foreign key (user_id) references "USER",
    constraint user_role_fk_2 foreign key (role_id) references ROLE,
    primary key (user_id, role_id)
);

INSERT INTO ROLE (id, role_name) VALUES (-1, 'ROLE_ADMIN');
INSERT INTO ROLE (id, role_name) VALUES (-2, 'ROLE_USER');

INSERT INTO USER_ROLE (user_id, role_id) VALUES (-1, -1);
INSERT INTO USER_ROLE (user_id, role_id) VALUES (-1, -2);
INSERT INTO USER_ROLE (user_id, role_id) VALUES (-2, -2);
----

[[account-id-hibernate-filter]]
== Colonne ACCOUNT_ID et filtre Hibernate

Quand une table contient une clé étrangère pointant vers la table `Account`, Celerio suppose que le contenu de cette table appartient à l'utilisateur représenté par la clé étrangère `account_id`.

Un filtre Hibernate est configuré pour s'assurer que cette table n'est chargée que par l'utilisateur courant.

Pour désactiver ce filtre :

1. Supprimer l'annotation `@Filter` de l'entité (implique de prendre le contrôle sur l'entité).
2. Désactiver le filtre programmatiquement via le helper `HibernateFilterContext` généré.
3. Désactiver globalement cette convention dans le fichier de configuration Celerio.

[[version-optimistic-locking]]
== Colonne VERSION et verrou optimiste

Si votre table a une colonne nommée `VERSION` dont le type est compatible avec l'annotation JPA `@Version`, Celerio suppose que vous voulez activer une stratégie de verrouillage optimiste.

En résultat, la propriété correspondante est annotée avec `@Version`.

Vous pouvez désactiver cette convention dans la configuration en utilisant l'attribut `version` du `columnConfig`.

[[many-to-many-inverse]]
== Many-to-many et attribut inverse

Quel côté de la relation `@ManyToMany` est marqué comme `inverse="true"` ?

Par convention, c'est le côté dont l'ordre de la colonne correspondante est le plus élevé sur la « table intermédiaire ». Vous pouvez surcharger cette convention en utilisant l'attribut `inverse` du `columnConfig` parent.

[[file-upload-download]]
== Upload et download de fichiers

Quand les propriétés suivantes sont présentes simultanément dans une entité, Celerio génère diverses méthodes helper pour faciliter la manipulation de fichiers de la couche web vers la couche persistance.

[cols="2,1,3", options="header"]
|===
|Propriétés
|Type
|Description

|_prefix_FileName
|String
|Nom du fichier

|_prefix_Size ou _prefix_Length ou _prefix_ContentLength
|Int
|Taille du fichier en octets

|_prefix_ContentType
|String
|Type MIME du fichier

|_prefix_Binary ou _prefix_Content ou _prefix_Blob
|Blob
|Contenu binaire du fichier
|===

Exemple SQL avec le préfixe `extract` (tel qu'utilisé dans celerio-angular-quickstart) :

[source,sql]
----
extract_binary         bytea,
extract_file_name      varchar(100),
extract_content_type   varchar(100),
extract_size           NUMERIC(11),
----

Cette convention permet d'uploader un fichier de manière transparente, de le sauvegarder dans la table correspondante, puis de le télécharger via une URL simple.

[[audit-in-entity]]
== Audit dans l'entité

Par convention, Celerio active la fonctionnalité d'audit si _l'une quelconque_ des propriétés suivantes est présente dans une entité.

[cols="2,1,2", options="header"]
|===
|Description
|Type
|Propriétés requises

|Auteur de la création
|String
|creationAuthor, creationBy, creePar

|Date de création
|Date
|creationDate, dateCreation

|Auteur de la dernière modification
|String
|lastModificationAuthor, lastModificationAt, derniereModificationPar, modifiePar

|Date de la dernière modification
|Date
|lastModificationDate, dateDerniereModification, derniereModification
|===

Exemple SQL :

[source,sql]
----
creation_date            timestamp,
creation_author          varchar(200),
last_modification_date   timestamp,
last_modification_author varchar(200),
----

Code Java généré :

[source,java]
----
@PrePersist
protected void prePersist() {
    if (AuditContextHolder.audit()) {
        setCreationAuthor(AuditContextHolder.username());
        setCreationDate(new Date());
    }
}

@PreUpdate
protected void preUpdate() {
    if (AuditContextHolder.audit()) {
        setLastModificationAuthor(AuditContextHolder.username());
        setLastModificationDate(new Date());
    }
}
----

[[audit-log-table]]
== Table de log d'audit

La fonctionnalité de log d'audit simple permet de tracker les événements de création/modification/suppression pour chaque entité et de les stocker dans une table dédiée.

Par convention, Celerio génère le code qui implémente cette fonctionnalité si les conditions suivantes sont remplies :

Une entité d'audit doit être présente. Son nom doit être :
* `Audit`
* `AuditLog`
* `AuditTrail`

Cette entité d'audit doit avoir les propriétés suivantes :

[cols="2,1,2", options="header"]
|===
|Propriétés requises
|Type
|Description

|author, auteur
|String
|Auteur de l'action

|event
|String
|Type d'événement (Creation, Modification, Deletion)

|stringAttribute1, attribute1, string1
|String
|Premier attribut (nom de l'entité)

|stringAttribute2, attribute2, string2
|String
|Deuxième attribut (id de l'entité)

|stringAttribute3, attribute3, string3
|String
|Troisième attribut (description des modifications)
|===

[[saved-search-table]]
== Table de recherches sauvegardées

Quand les colonnes suivantes sont présentes simultanément dans une table nommée `saved_search` ou `saved_search_form`, Celerio génère la fonctionnalité de sauvegarde de recherches.

[cols="2,1,3", options="header"]
|===
|Propriété
|Type
|Description

|formClass, formClassname
|String
|Nom complet du type du formulaire de recherche

|name
|String
|Nom donné à la recherche sauvegardée

|formContent
|byte[]
|Forme sérialisée de la recherche

|accountId
|String
|FK vers Account (non requis)
|===
